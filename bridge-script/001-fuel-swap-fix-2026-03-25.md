# Fuel Swap Bug Fix: `CurrencyNotSettled` Error

**Date:** 2026-03-25
**Branch:** `review/fuel-swap-pr10`

## Bug

The devnet deployment script (`pn start-devnet`) consistently failed during fuel swap tests with:

```
CurrencyNotSettled (selector: 0x5212cba1)
```

This error comes from Uniswap V4's `PoolManager` — it means a swap callback returned without fully settling all token deltas.

## Root Cause

**Flawed pool skip logic was causing the ETH/AZTEC pool to never be seeded.**

The old code in both `seed-pools.ts` and `index-devnet.ts` checked whether to skip seeding the ETH/AZTEC pool like this:

```ts
const pmFjBalance = await aztecToken.read.balanceOf([POOL_MANAGER])
if (pmFjBalance > 0n) {
  logger.info('ETH/AZTEC pool already seeded — skipping')
  return
}
```

The problem: `PoolManager`'s FeeJuice balance is **shared across ALL Uniswap V4 pools on Sepolia**, not just ours. If any other project on Sepolia had deposited FJ into any V4 pool, `pmFjBalance > 0n` would be true — and our specific ETH/AZTEC pool would be skipped even though it had zero liquidity.

The fuel swap route is: `USDC -> WETH -> ETH -> FeeJuice` (multi-hop through two pools). When the second hop (ETH/AZTEC) had no liquidity, the swap produced zero output but still had unsettled deltas, triggering `CurrencyNotSettled`.

## Solution

### 1. Remove all skip logic

`PoolSeeder.setup()` is already **idempotent** — it calls `pm.initialize()` inside a try/catch (skips if pool exists) then adds liquidity. Safe to call repeatedly. No skip logic needed.

### 2. Minimize ETH spend with math-verified params

Calculated exact V4 concentrated liquidity math from the contract's tick range `[69060, 115140]` at sqrtPrice ~10000:1:

**ETH/AZTEC pool (L=1e18):**
- Consumes: 0.00684 ETH + 68.4 FeeJuice
- Tradeable depth: ~0.022 ETH / ~68.4 FJ
- Fuel swap uses ~0.095 FJ (0.14% of depth)

**USDC/WETH pool (L=1e12):**
- Consumes: 35.59 USDC + 0.017 WETH
- Tradeable depth: ~159 USDC / ~0.017 WETH
- Fuel swap uses 0.02 USDC (0.01% of depth)

### Parameter changes

| Parameter | Before | After | Savings |
|---|---|---|---|
| `ETH_AZTEC_LIQUIDITY` | 25e18 | 1e18 | 25x less |
| `ETH_SEED` | 0.8 ETH | 0.01 ETH | 80x less |
| `FEE_MINT_COUNT` | 20 (20k FJ) | 1 (1k FJ) | 20x fewer txs |
| `ERC20_WETH_LIQUIDITY` | 8e12 | 1e12 | 8x less |
| `WETH_SEED` | 0.25 ETH | 0.02 ETH | 12.5x less |
| `ERC20_AMOUNT` | 1000 tokens | 100 tokens | 10x less |
| **Total ETH for seeding** | **~1.05 ETH** | **~0.03 ETH** | **35x less** |

## Files Modified

1. **`bridge-script/seed-pools.ts`** — Updated all pool params, env defaults, removed stale comments
2. **`bridge-script/index-devnet.ts`** — Updated pool params, mint count, ERC20 amount, fuel test comments
3. **`bridge-script/test-fuel-swap.ts`** — Reduced single-hop test amount (0.00001 -> 0.000001 ETH), fixed stale messages
4. **`l1-contracts/src/test/E2EPoolSeedAndSwap.t.sol`** — Aligned all Foundry test constants with new params

## No Contract Redeployment Needed

`UniswapFuelSwap`, `SwapBridgeRouter`, and `PoolSeeder` have no hardcoded liquidity params — sqrtPrice, ticks, and liquidity delta are all passed as call arguments. Only the script-side values changed.

## Key Takeaways

1. **Never use global PoolManager balances to infer per-pool state.** V4's PoolManager holds assets for ALL pools on the network. Use `PoolManager.getSlot0()` or pool-specific state queries instead.
2. **PoolSeeder.setup() is idempotent** — skip logic is unnecessary and actively harmful.
3. **V4 concentrated liquidity math**: At a 10000:1 price ratio with wide tick range, the minority-side token (FJ) has very small absolute depth per unit of liquidity. But L=1e18 still provides 68.4 FJ of depth — more than enough for fuel swaps.
4. **The old comments were wrong**: They claimed "25e18 liquidity gives ~0.19 FJ of tradeable depth" but the actual math shows 1e18 gives 68.4 FJ (so 25e18 would give ~1710 FJ). The error was off by ~9000x.
